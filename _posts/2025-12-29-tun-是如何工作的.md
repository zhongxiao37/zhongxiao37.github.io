## TUN 是什么

Clash Verge 的 TUN 模式（Network TUNnel）是一种基于虚拟网卡技术的网络流量接管方式。理解它的工作原理，最简单的方法是把它看作是操作系统里的一个**“网络流量黑洞”**，它会在底层拦截系统的所有网络请求。

要理解 TUN 模式，我们需要先对比一下普通的“系统代理”模式。

## 为什么需要 TUN 模式？

默认情况下，Clash Verge 使用的是系统代理（HTTP/SOCKS5 代理）。这种模式工作在 OSI 模型的应用层。
它的致命缺点是：它依赖于应用程序的自觉性。很多软件（比如命令行工具、大部分游戏、Windows UWP 应用，或者硬编码了直连逻辑的软件）根本不理会系统的代理设置，导致它们的流量直接绕过 Clash，出现“漏网之鱼”。

TUN 模式就是为了解决这个问题而诞生的。它工作在更底层的网络层（Layer 3），强制接管所有流量，实现真正的全局代理。

## TUN 模式的工作原理

当你开启 Clash Verge 的 TUN 模式时，后台会经历以下几个步骤：

1. 创建虚拟网卡 (Virtual Network Interface)
   Clash Verge 会向操作系统申请创建一个虚拟的网卡设备（在 Windows 上通常使用 wintun 驱动，在 macOS/Linux 上使用原生的 utun 或 tun 设备）。这个网卡在系统中看起来和真实的物理网卡（如 Wi-Fi 或以太网卡）没有区别。

2. 篡改系统路由表 (Modifying the Routing Table)
   这是最关键的一步。Clash 会修改操作系统的路由表，将这个新创建的虚拟网卡设置为默认网关，或者添加指向这个虚拟网卡的全路由规则（例如 0.0.0.0/0）。
   这样一来，系统里所有的应用程序（不管它是否支持代理）发出的 IP 数据包，都会被操作系统傻乎乎地直接扔进这个虚拟网卡里。

3. 用户态网络协议栈解析 (User-space TCP/IP Stack)
   进入虚拟网卡的都是最底层的原始 IP 数据包（Raw IP Packets）。Clash 作为一个应用层的代理软件，无法直接处理这些原始数据。
   因此，Clash 内部内置了一个用户态的 TCP/IP 协议栈（Clash Meta 内核通常使用 gVisor 或 lwIP）。这个协议栈负责将拦截到的 IP 数据包重新“组装”还原成原本的 TCP 连接或 UDP 数据流。

4. 规则匹配与分流 (Rule Matching)
   一旦流量被还原为 TCP/UDP 连接，Clash 就能拿到目标 IP 地址、端口甚至域名（如果配合 DNS 劫持/Fake IP）。此时，Clash 就会根据你配置的规则（比如 geosite:google、IP-CIDR 等）进行匹配：

Direct（直连）：如果规则判定为直连，Clash 会把流量从真实的物理网卡发出去。

Proxy（代理）：如果判定需要代理，Clash 会将流量加密封装，发送给你的代理服务器（节点）。

## TUN 模式的优缺点

优势：

- 100% 接管率：无论是命令行（如 curl, git）、虚拟机、Docker、还是游戏，统统都能被代理，不再需要为每个软件单独设置。
- 更好的 UDP 支持：对于支持 UDP 转发的节点，TUN 模式能更好地处理游戏的 UDP 流量和某些 P2P 流量。

劣势 / 注意事项：

- 需要管理员权限：因为需要创建虚拟网卡和修改系统核心路由表，所以每次开启 TUN 模式通常需要 UAC 提权（Windows）或 Root 权限（macOS/Linux）。不过 Clash Verge 提供了服务模式（Service Mode），安装后可以绕过每次的弹窗。

- 轻微的性能损耗：由于所有流量都需要经过一次“内核态 -> 用户态协议栈（gVisor/lwIP） -> 代理引擎”的转换，相比于纯粹的端口转发，会带来极少量的 CPU 资源消耗和延迟（在现代 CPU 上基本可以忽略不计）。

## 如何查看虚拟网卡和路由表

我开启了 Tailscale 之后，可以通过 `ifconfig | grep -A 3 "utun"` 查看

```bash
utun7: flags=8051<UP,POINTOPOINT,RUNNING,MULTICAST> mtu 1280
	options=6460<TSO4,TSO6,CHANNEL_IO,PARTIAL_CSUM,ZEROINVERT_CSUM>
	inet6 ffff::ffff:ffff:ffff:ffff%utun7 prefixlen 64 scopeid 0x23
	inet 100.100.100.100 --> 100.100.100.100 netmask 0xffffffff
	inet6 ffff::ffff:ffff:ffff:ffff prefixlen 48
	nd6 options=201<PERFORMNUD,DAD>

```

通过 `netstat -nr -f inet` 可以查看系统路由表，找到对应的`utun7`的记录，就可以确定流量的走向。

```bash
default            192.168.1.1        UGScIg                en0
default            link#35            UCSIg               utun7
100.64/10          link#35            UCS                 utun7
100.100.100.100/32 link#35            UCS                 utun7
100.100.100.100    link#35            UHWIi               utun7
224.0.0/4          link#35            UmCSI               utun7
255.255.255.255/32 link#35            UCSI                utun7
```

本来默认是走 `en0` 网卡，但是路由表的最长匹配原则，导致流量都被`utun7`接管了。

## 和 Tailscale 冲突怎么办

```yaml
tun:
  enable: true
  stack: system
  auto-route: true
  # 排除 Tailscale 的默认 CGNAT 网段，这部分流量系统会直接走原路由，不进 TUN
  route-exclude-address:
    - 100.64.0.0/10
```

## 和 OrbStack 的 orb.local 域名冲突怎么办

`.local`的域名是由 MacOS 底层接管了，所以没有所谓的 DNS 服务器。

通过 `scutil --dns` 可以看到由 mDNSResponder 处理了 `.local` 的域名。

```bash
resolver #4
  domain   : local
  options  : mdns
  timeout  : 5
  flags    : Request A records
  reach    : 0x00000000 (Not Reachable)
  order    : 300000
```

```yaml
dns:
  nameserver-policy:
    "+.orb.local": "system" # 1. 委派解析：遇到该域名，别走代理 DNS，交还给 macOS 系统的原生 DNS 处理
  fake-ip-filter:
    - "+.orb.local" # 2. 绕过污染：将该域名移出 Fake-IP 池，确保系统拿到的是 OrbStack 分配的真实局域网 IP

prepend-rules:
  - DOMAIN-SUFFIX,orb.local,DIRECT # 3. 流量直连：确保解析出真实 IP 后，流量直接在本地网卡流转，不被塞进代理隧道
```

## 为什么 TUN 模式需要 Fake DNS

虽然 TUN 可以接管所有流量，然后把 IP 包拆成 TCP/UDP 包，但是解出来也只知道 IP 地址，最上面的域名是在应用层里面。这里拦截之后，比如www.abc.com返回的fake ip 是 192.168.18.1，Clash 看到你的 TCP 包留到 192.168.18.1，就知道对应的域名是www.abc.com，就可以找到匹配的rule了。
